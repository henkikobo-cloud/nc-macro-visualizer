# v0.3.0 Plan: Adopting Nassi-Shneiderman Diagrams

> Review note (v0.3.x): N 番号の位置づけを「traceability data（補足情報）」から
> 「識別子（タイトルの一部）」に変更した。現場での実用性を踏まえた判断であり、
> v0.2.0 当初の方針から意図的に修正したものである。

## Purpose of This Document

This document records two things:

1. **v0.2.0 findings**: Mermaid + automatic layout has structural limits. This is not a Codex implementation defect. It is a limitation of the chosen representation.
2. **v0.3.0 plan**: Add Nassi-Shneiderman diagrams as the new primary beginner-facing representation. Move Mermaid flowcharts down to a developer-oriented supporting view.

This document is a decision record for after the v0.2.0 release. It is not an immediate implementation instruction for Claude Code.

## v0.2.0 Findings

### 1. Structural Limits of Mermaid Auto Layout

In v0.2.0, Mermaid with dagre auto layout was used as the main representation. The following mitigations were attempted:

- **Rule 1: long-distance GOTO connector nodes**: replace long edges with `jump_out` / `jump_in` connector pairs.
- **Rule 2: merge edge label consolidation**: reduce label collision at convergence nodes.
- **Revised Rule 2: decision node label suppression**: reduce the four-label collision around `はい` / `いいえ` / `次へ` / `合流`.
- **Rule 3: branch direction normalization**: emit `branch_false` before `branch_true`.

These mitigations removed long crossing edges caused by long-distance `GOTO`, but did not fully solve label collisions around `decision` nodes. The `IF ... THEN` merge pattern can still produce visually crowded labels.

### 2. Root Cause

This is not a builder implementation defect.

Mermaid / dagre has structural constraints:

- It cannot specify detour routes for long-distance edges.
- It cannot spread label positions for multiple edges into the same target.
- Final label placement is controlled by the layout algorithm, not by the source data.

Builder-side restructuring can reduce the problem, but cannot fully control the final Mermaid rendering.

### 3. Computer Science Background

Control-flow graphs are often classified as reducible or irreducible.

- Structured statements such as `IF`, `FOR`, `WHILE`, `BREAK`, and `CONTINUE` generally produce reducible CFGs.
- Arbitrary `GOTO` can produce irreducible CFGs.

NC macros often rely heavily on `GOTO`, so they can naturally produce irreducible CFGs. This is exactly the kind of graph that automatic layout engines such as Mermaid / dagre and Graphviz struggle with.

Therefore, making NC macro flowcharts consistently readable in Mermaid is fundamentally difficult. v0.2.0 established the limit of the Mermaid-first approach.

### 4. v0.2.0 Value

The following v0.2.0 work remains valuable and should be reused:

- `flowchart/` module: builder, models, terminology.
- `BeginnerFlow` data model.
- `schemas/beginner_flow.schema.json`.
- `--beginner` CLI flag.
- Regression tests.
- `web/` migration.

The issue is not the data model. The issue is the primary renderer.

### 5. Product Direction Impact

The v0.2.0 product rules remain valid:

- `N` numbers are identifiers shown at the start of beginner-facing titles because they are the shared language used on site.
- An `N` number must be paired with a meaning title, such as `N10: 値を計算する`.
- Lines without an `N` number use `line N` as the fallback identifier.
- The target user is a beginner or a PC-uncomfortable on-site worker.
- The tool is for understanding support, not machine behavior guarantee.

What changes in v0.3.0 is the primary beginner-facing representation.

Mermaid becomes a developer-oriented low-level CFG view. It is no longer the primary beginner-facing display.

## v0.3.0 Direction

### Primary Representation: Nassi-Shneiderman Diagram

Nassi-Shneiderman diagrams, also known as structograms or structured flowcharts, were introduced by Isaac Nassi and Ben Shneiderman in 1973.

They represent structured programs with nested boxes and no arrows.

### Why Adopt It

1. **No arrow overlap by design**

   There are no arrows, so crossing edges and edge-label placement conflicts disappear structurally.

2. **Alignment with product policy**

   The project aims to explain process meaning to beginners. Nested structured boxes are a better fit than a low-level CFG.

3. **Manufacturing-site familiarity**

   Nested boxes and table-like blocks are visually closer to manufacturing work instructions, QC process diagrams, and FMEA-style documents than graph-layout flowcharts.

4. **Low implementation cost**

   It can be rendered with static HTML/CSS using nested `div` elements and CSS grid. SVG is not required.

5. **Reuse of v0.2.0 model**

   `BeginnerFlow` is renderer-independent. A new renderer can be added without replacing the parser, analyzer, builder, schema, or existing tests.

### Handling GOTO

Nassi-Shneiderman diagrams do not naturally represent arbitrary `GOTO`. For this product, that is acceptable.

Recommended treatment:

- Short-distance GOTO, especially loop-like backwards jumps: interpret as `WHILE` / `DO-UNTIL` style boxes when the pattern is clear.
- Long-distance GOTO: show a terminal-style block such as `ここで N<target> へジャンプ`.
- Irreducible or unclear GOTO: show an annotation block such as `構造化が難しい箇所` and direct the user to the Mermaid developer view for details.

This keeps the 90% readable path simple and makes the remaining complex areas explicit.

## Alternatives Considered

### Candidate B: Structured Text View

Example:

```text
N50: ジャンプ -> N90 へ
N80: 値を計算する  #102 = 1
N90: 条件分岐  IF [#102 EQ 1]
       はい -> 値を入れる
       いいえ -> そのまま進む
```

Pros:

- No layout problem.
- Keeps source labels and meaning labels together.
- Easy to implement in Web UI.

Cons:

- Less visual impact for LP or demo use.

Decision:

- Good as a supporting text view.
- Not the primary v0.3.0 representation.

### Candidate C: Structured CFG Extraction + Limited Flowchart

This would parse NC macro as CFG, reduce loops and branches into structured statements, then draw a limited flowchart.

Pros:

- Keeps a flowchart-like look.
- Could reduce arrow overlap.

Cons:

- Heavy implementation.
- Irreducible GOTO remains a display problem.

Decision:

- Defer for v0.3.0.
- Prefer Nassi-Shneiderman first.

## v0.3.0 Migration Strategy

### Phase 1: Reposition Mermaid Renderer

After v0.2.0, update docs and UI copy:

- Mermaid renderer is a **developer-oriented low-level CFG view**.
- It is no longer the beginner-facing primary representation.
- README and app copy should warn that complex macros can be difficult to read in the current Mermaid-style view.
- Point users toward the v0.3.0 structured diagram plan.

### Phase 2: Add Nassi-Shneiderman Renderer

Proposed new files:

```text
src/nc_macro_visualizer/renderers/
├─ nassi_shneiderman.py
└─ structured_text.py

src/nc_macro_visualizer/flowchart/
└─ structurizer.py

web/
├─ nassi_shneiderman.css
└─ app.js
```

Do not change:

- `flowchart/builder.py`
- `flowchart/models.py`
- `flowchart/terminology.py`
- `schemas/beginner_flow.schema.json`
- Existing tests, except to add new tests.

### Phase 3: Web Default View Switch

At v0.3.0 release:

- Default beginner view: Nassi-Shneiderman diagram.
- Developer / detail view: Mermaid.
- Supporting view: structured text.
- Provide one-click switching between the three views.

## v0.3.0 Implementation Order (Draft)

Detailed execution instructions should be created separately at v0.3.0 start. Draft order:

1. Add `flowchart/structurizer.py`.
2. Add structured-tree models to `flowchart/models.py`.
3. Add `renderers/nassi_shneiderman.py`.
4. Add `renderers/structured_text.py`.
5. Add CLI flags such as `--nassi` and `--text`, while keeping `--beginner`.
6. Add Web view switching for the three views.
7. Update README and in-app guidance.

Run existing tests at every step.

## v0.3.0 Out of Scope

- Removing the Mermaid renderer.
- Removing `lp/demo/`.
- Fully implementing reducible CFG extraction.
- Pyodide integration.
- Machine-specific M code inference.
- Machining simulation.

## Open Questions Before v0.3.0

1. How deep should structural extraction go?
2. When should a GOTO be treated as too complex to structure?
3. How should `M98` / `G65` calls appear in nested boxes?
4. Should Web view switching use tabs, a dropdown, or segmented buttons?
5. Should Mermaid remain an expert view long term, or be phased out later?

## Overall Assessment

v0.2.0 was valuable because it established a renderer-independent beginner model and demonstrated the limits of Mermaid for NC macro understanding.

v0.3.0 should change the representation, not the core analysis engine.

The expected impact is limited to renderer and Web UI layers. Parser, analyzer, and existing `BeginnerFlow` generation should remain stable.
