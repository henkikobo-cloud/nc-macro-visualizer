# v0.2.0 Design Plan

> Review note: this was the initial v0.2.0 design plan. After technical review, the v0.2.0 implementation direction was narrowed. For execution, prefer `docs/claude_code_v0.2_refactor_instructions.md`: do not introduce `core/` in v0.2.0, add `flowchart/` beside the existing packages, define `schemas/beginner_flow.schema.json` first, and add beginner output as opt-in.

## 1. Current Structure Issues

v0.1.0 is useful as a CLI-first NC macro reading tool, but the structure still reflects a developer-facing analysis pipeline.

- The public entry point is effectively `nctool.py` / `nc_macro_visualizer.cli`.
- `parser`, `analyzer`, and `renderer` are coupled around reports for developers: `report.md`, `analysis.json`, and `flow.mmd`.
- Mermaid output currently uses NC line text and `N` labels as primary node labels. This is helpful for debugging, but not ideal for beginners.
- The current flowchart represents NC code connections more than the meaning of each process.
- `renderer` is singular, while the target structure expects `renderers`.
- There is no dedicated layer that translates parsed NC facts into beginner-facing concepts.
- Web demo code exists under `lp/demo/`, but it is not yet a first-class product entry point.
- The JavaScript demo has its own simplified parsing and translation logic, so terminology and behavior can drift from the Python engine.
- M code handling is conservative in Python, but beginner-facing display rules need a clearer product-level contract.

The main risk is changing the existing parser/analyzer/renderers too aggressively and breaking the stable v0.1.0 CLI tests. v0.2.0 should add a beginner-facing layer beside the current engine before moving or renaming major modules.

## 2. New Product Direction

NC Macro Visualizer v0.2.0 should be redesigned as an "NC macro understanding support app" for:

- NC beginners.
- Operators with limited macro reading experience.
- On-site workers who are not comfortable with command-line tools.
- People who need to understand process intent before asking an expert or checking the machine manual.

The product goal is:

> Translate NC macro code into a general, beginner-readable program flowchart.

The product must not claim to:

- Guarantee real machine behavior.
- Simulate machining.
- Reproduce PMC behavior.
- Infer machine-specific M code meanings.
- Validate whether a program is safe to run.

The main UI entry point should be a Web screen. CLI remains available for developers, regression tests, and internal checks.

## 4. Python Engine and Web UI Separation

The Python side should provide a stable analysis contract. The Web UI should render that contract, not reimplement parsing rules long term.

Python responsibilities:

- Parse NC text into structured facts.
- Extract labels, variables, controls, calls, M codes, warnings, and simple dependencies.
- Build beginner-facing flowchart nodes from analysis facts.
- Preserve uncertainty, especially for machine-specific M codes.
- Export stable JSON for Web consumption.

Web responsibilities:

- Let users choose or paste an NC file.
- Show a beginner-friendly flowchart.
- Show warnings in plain language.
- Show source-code details only as supporting information.
- Provide large, obvious controls for users who are not comfortable with PCs.
- Avoid developer terms unless the user opens details.

The Web UI should not become the source of truth for parsing. During v0.2.0, static JavaScript parsing can remain as a demo fallback, but the schema should match the Python output so it can later be replaced by generated JSON or a hosted API.

Proposed data flow:

```text
NC text
  -> core analysis result
  -> beginner flowchart model
  -> JSON / Mermaid / Markdown renderers
  -> Web UI rendering
```

## 5. Beginner Flowchart Specification

The beginner flowchart should be a general program flowchart, not an NC code connection diagram.

Node priorities:

1. Meaning of the process.
2. Plain-language condition or action.
3. Optional original NC line as detail.
4. Optional `N` label or source line for traceability.

Standard node types:

| Type | Shape | Use |
| --- | --- | --- |
| `start` | rounded terminator | Start of reading flow |
| `end` | rounded terminator | Program end or return |
| `process` | rectangle | Calculation, assignment, ordinary command |
| `decision` | diamond | `IF`, `IF ... GOTO`, `IF ... THEN`, `WHILE` |
| `call` | predefined process | `M98`, `G65`, subprogram or macro call |
| `machine_action` | rectangle with caution style | Known common M code |
| `needs_confirmation` | rectangle with warning style | Unknown or machine-specific M code |
| `jump` | connector or edge label | `GOTO` movement |
| `warning` | annotation | unresolved target, duplicate label, unsupported input |

Node label examples:

- `#500 = #100 + #101` -> `値を計算する`
- `IF [#100 GT 10] GOTO 80` -> `条件で分かれる`
- `GOTO 90` -> `指定した場所へ進む`
- `M98 P2000` -> `別のプログラムを呼び出す`
- `M30` -> `プログラムを終了する`
- `M123` -> `意味の確認が必要なMコード`

Edge labels:

- Sequential: `次へ`
- True branch: `はい`
- False branch: `いいえ`
- Jump: `指定場所へ`
- Loop body: `くり返す`
- Loop exit: `終了`
- Call: `呼び出し`

Traceability rules:

- `N` numbers and source line numbers must not be the main visual label.
- Keep original code in a collapsible detail area or small secondary text.
- JSON should retain `line_no`, `label`, and `source_text` for debugging and expert review.

## 6. Terminology Conversion Rules

Use plain Japanese terms in beginner-facing UI.

| Technical term | Beginner-facing term |
| --- | --- |
| variable | 変数 / 値の入れ物 |
| assignment | 値を入れる |
| expression | 計算式 |
| dependency | 値のつながり |
| condition | 条件 |
| branch | 分かれ道 |
| `IF ... GOTO` | 条件で別の場所へ進む |
| `IF ... THEN` | 条件が合うと処理する |
| `GOTO` | 指定した場所へ進む |
| `WHILE` | 条件の間くり返す |
| subprogram | 別プログラム |
| macro call | マクロ呼び出し |
| standard M code | 一般的な意味が知られているMコード |
| unknown M code | 意味の確認が必要なMコード |
| warning | 確認が必要な点 |

M code display rules:

- Known common M codes may show a common explanation.
- Unknown M codes must not be guessed.
- Unknown M codes should display: `意味の確認が必要なMコード`.
- Supporting message: `機械の説明書、PMC、ビルダー設定を確認してください。`
- Never display unknown M codes as if their behavior is guaranteed.

## 7. Web Demo Specification

The general-user entry point should be `web/index.html`.

Minimum v0.2.0 demo features:

- Open a sample NC macro immediately on page load.
- Allow file selection from the browser.
- Allow paste/edit in a text area.
- Provide one primary action: `解析する`.
- Display beginner flowchart as the main result.
- Display `確認が必要な点` near the top when warnings exist.
- Display `値のつながり`, `条件分岐`, and `Mコード` as secondary sections.
- Provide a simple Markdown memo download.
- Work as static HTML/CSS/JavaScript for GitHub Pages.

UI principles:

- Large buttons.
- Clear section titles.
- No command-line requirement.
- Avoid dense tables as the first view.
- Show original NC code only as detail, not as the main explanation.
- Use calm warning text instead of alarming error language.

Implementation note:

- The existing `lp/demo/` can be the starting point.
- Move it to `web/` after preserving the current demo behavior.
- Keep `lp/` as a promotional page and link to `../web/index.html` or the deployed Web URL.

## 8. CLI Positioning

The CLI remains supported, but it is not the main user experience.

CLI users:

- Developers.
- Maintainers.
- Test automation.
- Expert users who want raw artifacts.

CLI responsibilities:

- Read local NC files.
- Write `report.md`, `analysis.json`, and `flow.mmd`.
- Optionally write a beginner-flow JSON file in v0.2.0, such as `beginner_flow.json`.
- Preserve existing commands so v0.1.0 users and tests do not break.

Do not remove:

- `nctool.py`
- `nc_macro_visualizer.cli.main:main`
- Current output names
- Existing test expectations

CLI documentation should move below Web usage in README.

3. Implement a builder that converts existing `AnalysisResult` into a beginner flowchart model.
4. Add tests for beginner node titles, unknown M code wording, and branch edge labels.
5. Add JSON rendering for beginner flowchart output.
6. Keep current Mermaid renderer unchanged, then add a new beginner Mermaid renderer separately.
7. Move or copy `lp/demo/` to `web/` after tests pass.
8. Update README so Web is the first entry point and CLI is developer-facing.
9. Add CLI option only if needed, for example `--beginner-flow`, without changing default outputs.
10. Update release notes and limitations for v0.2.0.

Possible v0.2.0 outputs:

```text
output/
├─ report.md
├─ analysis.json
├─ flow.mmd
└─ beginner_flow.json
```

Do not do in v0.2.0:

- Full NC interpreter.
- Full expression AST unless needed by tests.
- Hosted backend.
- Machine-specific M code inference.
- Toolpath or machining simulation.

## 10. Migration Plan Without Breaking Existing Tests

Step-by-step migration:

1. Add new modules only. Do not rename existing packages yet.
2. Add tests for new beginner flowchart behavior.
3. Keep all current tests passing:
   - `tests/test_parser.py`
   - `tests/test_analyzer.py`
   - `tests/test_renderer.py`
   - `tests/test_cli.py`
4. Keep current CLI output files unchanged.
5. Add new output behind an option or as an additional file only after tests cover it.
6. Introduce `renderers/` as compatibility wrappers if the plural directory is needed:
   - `renderers.markdown` imports from `renderer.markdown`
   - `renderers.mermaid` imports from `renderer.mermaid`
   - `renderers.json_renderer` imports from `renderer.json_renderer`
7. Move implementation from old packages to new packages only after wrapper tests exist.
8. Move Web demo from `lp/demo/` to `web/` in a separate change.
9. Update README and docs after the code path is stable.
10. Run both test commands before release:

```bash
python3 -m unittest discover -s tests
.venv/bin/python -m pytest
```

Rollback-friendly rule:

- Every structural move should keep an import compatibility layer until at least one release after v0.2.0.
